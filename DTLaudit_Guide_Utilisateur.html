DTLaudit — Guide utilisateur

Ce guide explique comment utilisér DTLaudit pour contrôler rapidement l’état d’un projet ou d’un ensemble de projets locaux.

WindowsPythonLecture seule

Objectif

DTLaudit sert à obtenir une vue d’ensemble : quels projets sont des dépôts Git, lesquels ont un README, lesquels possèdent une release GitHub, quels dossiers générés encombrent le dépôt, et quels fichiers sont présents seulement dans certains projets.

DTLaudit ne modifie pas les projets analysés. Il observe, produit un rapport, puis s’arrête.

Démarrer

Lancement graphique

Si DTLaudit est lancé sans argument, une fenêtre permet de choisir le dossier ou le projet à analyser.

Lancement en ligne de commande

La ligne de commande reste utile pour répéter toujours le même audit.

python DTLaudit.py --suite "D:\Documents\Mes sites Web\outils"

Auditer une suite de projets

  1. Choisir le dossier qui contient les projets.
  2. Lancer l’audit en mode suite.
  3. Ouvrir le rapport HTML généré.
  4. Lire d’abord le tableau de synthèse, puis les blocs par projet.

Auditer un projet unique

Le mode projet se concentre sur un seul dossier. Il est utile lorsqu’un dépôt précis paraît incohérent ou avant une publication.

python DTLaudit.py --project "D:\Documents\Mes sites Web\outils\GitHubMenu"

Lire le rapport

Tableau de synthèse

Il indiqué pour chaque projet la présence Git, GitHub, README, release, branche active et modifications locales.

Observations

Les observations mettent en avant les écarts significatifs : fichiers volumineux, dossiers générés, documents absents ou différences entre projets.

Matrice des fichiers

Elle aide à repérer les fichiers présents dans un seul projet ou dans certains projets seulement.

Métadonnées GitHub

Si GitHub CLI (`gh`) est installé et authentifié, DTLaudit complète le rapport avec des informations GitHub : visibilité du dépôt, dernière release, issues et pull requests ouvertes.

Si `gh` n’est pas disponible, l’audit local reste exploitable.

Conseils pratiques

  • Lancer DTLaudit avant une série de publications GitHub.
  • Surveiller les dossiers `build`, `dist` et cachés Python.
  • Comparer régulièrement les README, guides et manuels entre projets.
  • Utiliser le rapport HTML comme tableau de bord de contrôle ponctuel.
